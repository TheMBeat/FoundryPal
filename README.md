# FoundryPal Monorepo Foundation

Technical foundation for a self-hosted Foundry VTT mobile companion with strict bridge boundaries.

## Architecture decisions

1. **Monorepo with one shared contract package and two apps**
   - `packages/shared` owns all DTOs, commands, and transport message schemas.
   - `apps/foundry-module` is the trusted server-side bridge module running inside Foundry.
   - `apps/companion-web` is a mobile-first React companion client using only shared contracts.
2. **Trust boundary is the Foundry module**
   - Client is untrusted.
   - Every command is schema validated with Zod.
   - Every command resolves actor by authenticated session actor list.
   - No arbitrary patch endpoint exists.
3. **Transport abstraction first**
   - `BridgeTransport` abstraction allows moving from in-memory transport to WebSocket/REST hybrid without changing command/DTO logic.
4. **DTO/read-model mapping layer**
   - dnd5e/Foundry documents are mapped to explicit DTOs. Raw documents are never sent.
5. **MVP auth for local self-hosting**
   - Session token generated by bridge module (`SessionService`) with TTL.
   - Session includes allowed actor IDs derived from Foundry ownership at issue time.
   - Session is required for command execution.

## Monorepo structure

- `packages/shared`: Zod schemas + TypeScript types for DTOs, commands, transport envelopes.
- `apps/foundry-module`: Foundry module manifest + bootstrap + bridge services + command handlers + tests.
- `apps/companion-web`: iPhone-first companion app with bottom tabs for character, actions, inventory, spells, and chat.
- `infra`: env examples.

## Companion webapp contracts used

The web app consumes only shared schemas/contracts from `@foundrypal/shared`:
- DTOs: `CompanionUserSession`, `CompanionActorSummary`, `CompanionActorDetail`, `CompanionChatMessage`.
- Commands: `updateHP`, `spendResource`, `useItem`, `castSpell`, `sendChatMessage`.
- Transport envelopes: `session`, `snapshot`, `event`, `command`, `commandResult`, `error`.

## Security boundary notes

### Secure now
- Runtime schema validation for all command payloads.
- Server-side authorization on each command.
- Actor access locked to `session.actorIds`.
- Command-specific handlers for mutating operations (`updateHP`, `spendResource`, `useItem`, `castSpell`, `sendChatMessage`).
- Basic chat rate-limiting hook (`ChatRateLimiter`).

### Known MVP limitations
- Session store is in-memory; restarting Foundry invalidates sessions.
- Session issuance endpoint transport/auth handshake is not yet wired to a real HTTP/WebSocket server.
- Chat sanitization relies on Foundry behavior; additional sanitization may be needed depending on deployment.
- Ownership snapshot is fixed at session issue time; permission changes mid-session require reissue.
- Actions tab currently displays compatible actions but intentionally keeps execution controls disabled until a dedicated action command contract exists.

## Development

### Prerequisites
- Node 22+
- pnpm 9+

### Install

```bash
corepack enable
pnpm install
```

### Scripts

```bash
pnpm build
pnpm typecheck
pnpm test
pnpm dev
```

### Companion webapp

```bash
pnpm --filter @foundrypal/companion-web dev
```

Set bridge socket URL (optional):

```bash
VITE_BRIDGE_WS_URL=ws://localhost:3001/bridge pnpm --filter @foundrypal/companion-web dev
```

### Docker

```bash
docker compose up --build
```

This currently installs dependencies and runs test suite in-container.

## Environment variables

Defined in `infra/.env.example`:
- `FOUNDRYPAL_SESSION_TTL_MS`
- `FOUNDRYPAL_CHAT_RATE_MAX`
- `FOUNDRYPAL_CHAT_RATE_WINDOW_MS`

## Foundry module install notes

1. Build module: `pnpm --filter @foundrypal/foundry-module build`
2. Copy `apps/foundry-module/module.json` and `apps/foundry-module/dist` into your Foundry Data modules directory.
3. Enable `foundrypal-bridge` module in Foundry v13 with dnd5e.
